# Дизайн улучшений продукта

Документ описывает следующий слой развития Vamos Subscription Tracker после перехода на серверный режим.

## 0. Продуктовая модель и опыт оператора

### Назначение продукта

Vamos Subscription Tracker - это операционный кабинет для учета абонементов, занятий, оплат и работы преподавателей.

Продукт должен отвечать на ежедневные вопросы оператора и владельца:

- кто из участников пришел на занятие;
- по какому абонементу списано занятие;
- сколько занятий осталось у участника;
- к какому преподавателю пошел участник;
- какие оплаты были внесены;
- сколько заработали преподаватели;
- какие действия выполнялись в системе и кем.

Система не должна смешивать активность абонемента и оплату. Абонемент может быть активен даже при частичной оплате, а оплата сама по себе не должна автоматически менять статус абонемента.

### Основные сущности

`Participant` - участник или ученик.

Участник является центральной клиентской сущностью. Вокруг него строится карточка: персональные данные, текущий абонемент, история абонементов, посещений и оплат.

`MembershipType` - тип абонемента.

Это шаблон продукта: название, количество занятий, цена и срок действия. Оператор выбирает тип абонемента, чтобы не вводить параметры вручную каждый раз.

`Membership` - конкретный абонемент участника.

Абонемент создается на основе типа абонемента и привязывается к участнику. В нем фиксируются общее количество занятий, остаток, цена, даты начала и окончания, а также статус.

`Visit` - факт списания занятия.

Посещение связывает участника, абонемент и преподавателя. Именно посещения являются основой для расчета заработка преподавателей. Если занятие списано ошибочно, оно не удаляется, а переводится в статус возвращенного.

`Teacher` - преподаватель.

Преподаватель выбирается при списании занятия. В финансовой аналитике по преподавателю считаются только неотмененные занятия.

`Payment` - оплата.

Оплата фиксирует факт поступления денег от участника. Она не меняет автоматически активность абонемента и не влияет на остаток занятий.

`Operator` - пользователь системы.

Оператор входит в приложение, выполняет действия и получает доступ в рамках своей роли.

`AuditLog` - журнал действий.

Журнал фиксирует, кто, когда и что сделал в системе: создал участника, выдал абонемент, списал занятие, вернул занятие, добавил оплату или изменил настройки.

### Связи сущностей

Создание абонемента:

```text
MembershipType -> Membership -> Participant
```

Тип абонемента задает правила продукта, а конкретный абонемент применяет их к участнику.

Списание занятия:

```text
Participant + Membership + Teacher -> Visit
```

При списании занятия система:

- проверяет активный абонемент;
- проверяет остаток занятий;
- создает посещение;
- уменьшает остаток занятий;
- связывает занятие с преподавателем;
- обновляет финансовую аналитику преподавателя.

Возврат занятия:

```text
Visit.is_cancelled = true
Membership.remaining_lessons += 1
```

Возврат должен выполняться только через отмену конкретного посещения. Нельзя возвращать занятие ручным изменением остатка без связи с исходным списанием.

### Финансовая логика

Финансы разделяются на две независимые плоскости.

Оплаты участников:

```text
Payment = сумма денег, внесенная участником
```

Заработок преподавателей:

```text
teacher_earning = количество неотмененных занятий * стоимость занятия
```

Стоимость занятия считается от абонемента:

```text
lesson_price = membership.price / membership.total_lessons
```

Пример:

```text
Абонемент: 12 000 ₽ / 8 занятий
Стоимость занятия: 1 500 ₽
Преподаватель провел 4 занятия
Заработок преподавателя: 6 000 ₽
```

Если занятие возвращено, оно не должно учитываться в заработке преподавателя.

### Основной опыт оператора

Оператор начинает работу с главной панели.

На главной панели он должен быстро понимать текущее состояние:

- сколько активных абонементов;
- какие абонементы скоро закончатся;
- какие оплаты были последними;
- какие занятия были списаны последними;
- сколько проведено занятий;
- какой доход школы;
- сколько начислено преподавателям.

Основной сценарий списания занятия:

1. Оператор открывает раздел “Участники”.
2. Находит участника по ФИО или телефону.
3. Открывает карточку участника или использует действие из таблицы.
4. Нажимает “Списать занятие”.
5. Выбирает преподавателя.
6. Подтверждает списание.
7. Система создает посещение и уменьшает остаток занятий.
8. В истории участника появляется новое занятие.
9. В финансах обновляется заработок преподавателя.

Сценарий нового участника:

1. Оператор создает участника.
2. Создает ему абонемент.
3. При необходимости фиксирует оплату.
4. Дальше участник попадает в обычный поток списания занятий.

Сценарий ошибки:

1. Оператор или администратор видит ошибочно списанное занятие.
2. Открывает историю посещений.
3. Нажимает “Вернуть занятие”.
4. Система помечает посещение как возвращенное.
5. Остаток занятий увеличивается.
6. Заработок преподавателя пересчитывается без возвращенного занятия.

### Продуктовая ценность

Приложение закрывает три ключевые зоны:

- операционный учет: участники, абонементы, остатки и посещения;
- финансовая прозрачность: оплаты, доход школы и начисления преподавателям;
- контроль ошибок: возвраты занятий, отмены оплат и журнал действий.

С точки зрения продукта Vamos Subscription Tracker должен восприниматься не как бухгалтерская система и не как CRM, а как ежедневный рабочий инструмент школы или студии для управления абонементами и занятиями.

## 1. Финансы

### Цель

Сделать страницу финансов понятной владельцу школы и оператору: сколько продано, сколько реально получено, сколько уже отработано занятиями, сколько нужно выплатить преподавателям и сколько остается школе.

Верхние KPI-карточки:

- Продажи абонементов.

- преподаватель;
- тип абонемента;

- переключатель “по дате оплаты / по дате занятия / по дате продажи”.

### Таблица “Преподаватели”

Колонки:

- преподаватель;
- проведено занятий;
- стоимость занятий;
- процент преподавателя;
- к выплате;
- доход школы;
- средняя стоимость занятия;
- последняя дата занятия.

Строка раскрывается и показывает занятия:

- дата;
- участник;
- абонемент;
- стоимость занятия;
- процент;
- выплата преподавателю;
- доход школы;
- статус: проведено / возвращено.

### Таблица “Оплаты”

Колонки:

- оператор;
- статус.

## 2. Роли пользователей

### Цель

Разделить доступы, чтобы финансовые и административные действия были доступны не всем операторам.

### Роли MVP

Администратор:

- управление пользователями;
- управление преподавателями;
- управление типами абонементов;
- просмотр и экспорт финансов;
- отмена оплат;
- отмена абонементов;
- возврат занятий;
- все права оператора.

Оператор:

- участники;
- создание абонементов;
- списание занятий;
- добавление оплат;
- просмотр базовой панели;
- просмотр карточки участника.

Финансист:

- просмотр финансов;
- просмотр оплат;
- экспорт отчетов;
- без права менять участников и занятия.

### Изменения в backend

Добавить в `Operator` поле:

```text
role = admin | operator | finance
```

Добавить dependency-проверки:

- `require_admin`;
- `require_finance_access`;
- `require_operator_access`.

### Изменения в frontend

В `AuthProvider` хранить роль оператора.

Скрывать пункты меню и действия:

- “Пользователи” только admin;
- “Финансы” admin и finance;
- отмена оплаты только admin;
- отмена абонемента только admin;
- изменение процента преподавателя только admin.

Важно: скрытие в UI не заменяет backend-проверки.

## 3. Политика паролей

### Цель

Сделать доступ операторов предсказуемым и безопасным без усложнения входа для ежедневной работы.

Парольная политика должна защищать от простых и повторяющихся паролей, но не должна превращать авторизацию в отдельный сложный продукт.

### Требования к паролю

Минимальные требования:

- длина не менее 8 символов;
- минимум одна латинская буква;
- минимум одна цифра;
- минимум один специальный символ;
- пароль не должен совпадать с логином;
- пароль не должен состоять только из пробелов;
- пароль не должен быть одним из очевидных вариантов: `password`, `12345678`, `qwerty123`, `admin123`, `operator`.

Рекомендуемые требования для production:

- длина 10-12 символов и больше;
- наличие букв в разных регистрах;
- уникальный пароль для каждого оператора;
- запрет повторного использования последнего пароля после ручной смены.

### Создание и изменение пароля

Администратор создает пользователя и задает первичный пароль.

При создании пользователя:

- backend валидирует пароль;
- frontend показывает понятные требования до отправки формы;
- пароль не отображается в списке пользователей;
- пароль не возвращается из API;
- в журнал действий пишется факт создания пользователя, но не сам пароль.

При изменении пароля:

- пароль можно изменить только через отдельное поле;
- пустое поле пароля при редактировании означает “не менять пароль”;
- в журнал действий пишется факт изменения пароля без значения пароля;
- текущий пользователь не должен иметь возможность случайно отключить сам себя.

### Хранение паролей

Пароли нельзя хранить в открытом виде.

Backend должен хранить только хеш пароля:

```text
password_hash
```

Текущий подход:

```text
PBKDF2-SHA256 + salt + iterations
```

Требования:

- у каждого пароля должен быть уникальный salt;
- сравнение пароля должно выполняться через защищенное сравнение;
- `password_hash` не должен возвращаться в API;
- пароль не должен попадать в audit log;
- пароль не должен попадать в frontend state, кроме формы ввода до отправки.

### Ошибки входа

Сообщение об ошибке входа должно быть нейтральным:

```text
Неверный логин или пароль
```

Не нужно отдельно сообщать:

- что логин существует;
- что пароль неверный;
- что пользователь отключен.

Это снижает риск перебора пользователей.

### Блокировка и ограничения

Для MVP достаточно:

- не показывать технические детали ошибки;
- логировать успешные входы;
- логировать критичные действия пользователя.

Следующий уровень безопасности:

- ограничение количества попыток входа;
- временная блокировка после нескольких ошибок;
- журнал неуспешных попыток входа;
- принудительная смена временного пароля при первом входе;
- срок действия пароля для администраторов.

### Системные доступы

Системные или аварийные учетные записи не должны смешиваться с обычными ролями продукта.

Правила:

- системный доступ не отображается в списке пользователей;
- системный доступ не редактируется через обычный UI;
- системный доступ используется только для тестовой среды или аварийного доступа;
- на production такие учетные записи должны быть отключены, если нет отдельного согласованного процесса.

Для текущей схемы:

- production не должен иметь скрытого `root`;
- `/v1` может иметь скрытого `root` для тестирования;
- любые системные пароли должны храниться и передаваться отдельно от обычной пользовательской документации.

## 4. Журнал действий

### Цель

Дать владельцу прозрачность: кто, когда и что сделал в системе.

### Модель AuditLog

Поля:

- id;
- operator_id;
- operator_name;
- action;
- entity_type;
- entity_id;
- entity_label;
- before_json;
- after_json;
- created_at;

### События MVP

Логировать:

- вход оператора;
- создание/редактирование участника;
- создание абонемента;
- заморозка/разморозка/отмена абонемента;
- списание занятия;
- возврат занятия;
- добавление оплаты;
- отмена оплаты;
- создание/редактирование преподавателя;
- изменение процента преподавателя;
- создание/редактирование типа абонемента;
- создание/отключение пользователя.

### API

```text
GET /audit-logs
```

Фильтры:

- date_from;
- date_to;
- operator_id;
- action;
- entity_type;
- entity_id.

### UI

Новый раздел “Журнал” в боковом меню для admin.

Таблица:

- дата и время;
- оператор;
- действие;
- объект;
- краткое описание.

Карточка события:

- кто сделал;
- когда;
- что изменилось;
- значения до;
- значения после.

### Принципы

- Лог нельзя редактировать через UI.
- Лог нельзя удалять обычным оператором.
- Для персональных данных в журнале показывать только нужные поля.
- Для ошибок логировать только бизнес-действия, не технический шум.

### Минимальный чеклист перед релизом

Перед переносом изменения из `/v1` в production нужно проверить:

- вход под каждой ролью;
- доступность главной панели;
- создание участника;
- создание абонемента;
- списание занятия с выбором преподавателя;
- возврат занятия;
- добавление оплаты;
- отмену оплаты администратором;
- финансовую сводку;
- заработок преподавателей;
- журнал действий;
- отсутствие скрытых тестовых пользователей на production;
- сохранность существующей production-базы после деплоя.

### Автоматизированное покрытие

Часть матрицы должна быть закреплена автоматическими тестами, чтобы правила не зависели только от ручной проверки.

Минимальный набор backend-тестов:

- парольная политика принимает сильные пароли и отклоняет слабые;
- создание пользователя невозможно со слабым паролем;
- изменение пароля невозможно на слабый пароль;
- системный `root` скрыт из управления пользователями;
- роли `admin`, `operator`, `finance` проходят только разрешенные проверки доступа;
- возврат занятия нельзя выполнить дважды;
- возвращенное занятие не учитывается в заработке преподавателя.

## 6. Рекомендуемый порядок реализации

1. Роли backend.
2. Роли frontend.
3. AuditLog backend.
4. Журнал действий UI.
5. Финансы UI.
6. Экспорт финансов в CSV/XLSX.

Такой порядок снижает риск: сначала закрываются права доступа, потом появляется аудит действий, затем улучшается отчетность.
